# Global Rules

> **Purpose:** System-wide standards that ALL modules must follow.
> **Created:** 2026-01-17
> **Status:** Planning Document

---

## Core Philosophy

**Use the best tool for the job.**

- System handles content creation (tones, prompts, palettes) - it has your values built in
- Human handles novel strategic decisions - different job, not fallback
- Pass references, not data - lean payloads, fetch on demand
- Self-healing - system creates missing entries using its own capabilities

---

## System Architecture (Kitchen Analogy)

| Concept | Kitchen | System |
|---------|---------|--------|
| Data (repos) | Ingredients in pantry | JSON files in `/repos/` |
| Functions | Actions (crack, whisk, measure) | Single atomic operations |
| Steps | Numbered recipe instructions | Sequence of functions within ONE module |
| Module config | Recipe card for one station | Defines functions + steps for that module |
| System config | Head chef's menu/orders | Chains that orchestrate across modules |
| Orchestrator | Head chef | Reads system.config, tells modules what to do |

### How It Works

1. **Modules are independent stations** - Each module has its own "cupboard" (available functions) and "recipes" (steps that combine functions)

2. **Steps stay within one module** - Like making icing separately before adding to cake. The icing module knows how to mix its ingredients.

3. **Chains orchestrate across modules** - The head chef (Orchestrator) reads the system.config and tells each station what to make and in what order.

4. **Pass references, not ingredients** - Instead of passing the actual tone data, pass `refs.tone = "formal-corporate"`. Each module fetches what it needs from the pantry (repos).

## JSON Data Rules

### File Structure

All repo data uses **one file per entry**:

```
/repos/{repo-name}/
  {entry-id}.json          ← Individual entry
  {entry-id}.json
  _index.json              ← Librarian's cached index (auto-generated)
```

### Entry ID Rules

| Rule | Example | Why |
|------|---------|-----|
| Lowercase | `gpt-4-turbo` not `GPT-4-Turbo` | Consistent lookups |
| Kebab-case | `formal-corporate` not `formal_corporate` | URL-safe, readable |
| No spaces | `business-formal` not `business formal` | File system safe |
| Descriptive | `friendly-casual-v2` not `fc2` | Self-documenting |

### Required Fields (All Entries)

Every repo entry MUST have:

```json
{
  "id": "entry-id",           // Matches filename (without .json)
  "name": "Human Readable",   // Display name
  "created": "2026-01-17",    // ISO date
  "updated": "2026-01-17",    // ISO date
  "status": "active",         // active | deprecated | draft
  "version": "1.0"            // Semantic version
}
```

### Optional Standard Fields

```json
{
  "description": "What this entry is for",
  "tags": ["tag1", "tag2"],           // For Librarian search
  "domain": "business",               // Category/domain
  "type": "text",                     // Entry type within repo
  "deprecated_by": "new-entry-id",    // If status: deprecated
  "notes": "Internal notes"
}
```

### Index File (_index.json)

Auto-generated by Librarian. Contains searchable metadata:

```json
{
  "repo": "tones",
  "updated": "2026-01-17T14:30:00Z",
  "count": 42,
  "entries": [
    {
      "id": "formal-corporate",
      "name": "Formal Corporate",
      "tags": ["professional", "business"],
      "domain": "business",
      "status": "active"
    }
  ]
}
```

**Never edit _index.json manually** - Librarian rebuilds it on repo changes.

## Naming Conventions

### Modules

| Type | Pattern | Example |
|------|---------|---------|
| Service Module | PascalCase | `PromptBuilder`, `FileManager` |
| System Service | PascalCase | `Writer`, `Logger`, `FailSys` |
| Integration | PascalCase + suffix | `CanvaIntegration`, `CompaniesHouseIntegration` |

### Functions (Bricks)

| Pattern | Example | Why |
|---------|---------|-----|
| camelCase | `loadTemplate()` | JavaScript standard |
| Verb first | `getEntry()`, `validateSchema()` | Action-oriented |
| Descriptive | `resolveVariables()` not `resolve()` | Self-documenting |

### Steps

| Pattern | Example |
|---------|---------|
| Module.stepName | `TokenManager.validateAndTrack` |
| camelCase step | `PromptBuilder.compileWithContext` |

### Chains

| Pattern | Example |
|---------|---------|
| kebab-case | `brand-kit`, `create-tone-entry` |
| Descriptive | `visual-refresh`, `legal-update` |

### Files

| Type | Pattern | Example |
|------|---------|---------|
| Module file | kebab-case.js | `prompt-builder.js` |
| Config | name.config.json | `module.config.json` |
| Rules | name.rules.json | `retry.rules.json` |
| Schema | name.schema.json | `tone.schema.json` |
| Test | name.test.js | `prompt-builder.test.js` |

## Config File Standards

### module.config.json

Every module has a config file listing its functions and steps:

```json
{
  "module": "TokenManager",
  "version": "1.0.0",
  "description": "Token estimation, budgets, and usage tracking",

  "functions": [
    {
      "name": "estimateTokens",
      "description": "Estimate token count for content",
      "input": ["content", "model"],
      "output": "tokenEstimate"
    },
    {
      "name": "checkBudget",
      "description": "Check if request is within budget",
      "input": ["brandId", "tokenEstimate"],
      "output": "budgetStatus"
    },
    {
      "name": "recordUsage",
      "description": "Record token usage after API call",
      "input": ["jobId", "actualTokens", "cost"],
      "output": "usageRecord"
    }
  ],

  "steps": [
    {
      "name": "validateAndTrack",
      "description": "Full token validation and tracking flow",
      "sequence": ["estimateTokens", "checkBudget", "recordUsage"]
    }
  ],

  "config": {
    "defaultModel": "gpt-4-turbo",
    "warningThreshold": 0.75,
    "blockThreshold": 1.0
  }
}
```

### system.config.json

Master index of all modules and chains:

```json
{
  "system": "NCE",
  "version": "1.0.0",

  "modules": {
    "TokenManager": "./modules/token-manager/module.config.json",
    "PromptBuilder": "./modules/prompt-builder/module.config.json",
    "AICaller": "./modules/ai-caller/module.config.json"
  },

  "chains": {
    "brand-kit": {
      "description": "Generate full brand kit",
      "steps": [
        "Builders.scaffoldBrand",
        "ModelManager.selectWithFallback",
        "TokenManager.validateAndTrack",
        "AICaller.callWithRetry"
      ]
    }
  }
}
```

### Rules Files (*.rules.json)

For handlers that need configurable behavior:

```json
{
  "retry": {
    "maxAttempts": 3,
    "backoff": "exponential",
    "baseDelay": 1000
  },
  "circuitBreaker": {
    "failureThreshold": 3,
    "windowMs": 300000,
    "cooldownMs": 21600000
  }
}
```

## Error Handling Standards

### Error Object Structure

All errors must include:

```json
{
  "error": true,
  "code": "TONE_NOT_FOUND",
  "message": "Tone 'formal-corporate' not found in repos",
  "module": "PromptBuilder",
  "function": "loadContext",
  "jobId": "job_20260117_143052",
  "timestamp": "2026-01-17T14:30:52Z",
  "context": {
    "toneId": "formal-corporate",
    "repo": "tones"
  },
  "recoverable": true,
  "suggestedAction": "librarian_recommend"
}
```

### Error Codes

| Prefix | Category | Example |
|--------|----------|---------|
| `NOT_FOUND_` | Missing data | `NOT_FOUND_TONE`, `NOT_FOUND_MODEL` |
| `VALIDATION_` | Schema/input errors | `VALIDATION_SCHEMA`, `VALIDATION_INPUT` |
| `API_` | External API errors | `API_TIMEOUT`, `API_RATE_LIMIT` |
| `AUTH_` | Authentication errors | `AUTH_EXPIRED`, `AUTH_INVALID` |
| `BUDGET_` | Cost/token limits | `BUDGET_EXCEEDED`, `BUDGET_WARNING` |
| `SYSTEM_` | Internal errors | `SYSTEM_DISK_FULL`, `SYSTEM_CONFIG` |

### Recoverable vs Non-Recoverable

| Recoverable | Action | Example |
|-------------|--------|---------|
| `true` | FailureHandler can retry/recommend | Missing tone, API timeout |
| `false` | Requires human intervention | Invalid config, disk full |

## Logging Standards

### Log Levels

| Level | When | Example |
|-------|------|---------|
| `debug` | Development detail | Variable values, flow traces |
| `info` | Normal operations | Job started, step completed |
| `warn` | Recoverable issues | Budget at 75%, retry attempt |
| `error` | Failures | API error, validation failed |
| `fatal` | System halt | Config invalid, disk full |

### Log Entry Structure

```json
{
  "level": "info",
  "timestamp": "2026-01-17T14:30:52.123Z",
  "module": "AICaller",
  "function": "callWithRetry",
  "jobId": "job_20260117_143052",
  "message": "API call successful",
  "data": {
    "model": "gpt-4-turbo",
    "tokens": 1523,
    "latency": 2341
  }
}
```

### What to Log

| Always Log | Never Log |
|------------|-----------|
| Job start/end | API keys |
| Step transitions | Full prompt content (too large) |
| Errors (all) | User passwords |
| Retries | Raw API responses (use temp) |
| Budget warnings | |

## Job Context Standards

### Required Fields

Every job context MUST have:

```json
{
  "jobId": "job_20260117_143052_brandkit_abc123",
  "chainName": "brand-kit",
  "brandId": "acme",
  "created": "2026-01-17T14:30:52Z",
  "status": "in_progress"
}
```

### Job ID Format

```
job_{date}_{time}_{chain}_{random}
job_20260117_143052_brandkit_abc123
```

| Part | Format | Example |
|------|--------|---------|
| Prefix | `job_` | `job_` |
| Date | YYYYMMDD | `20260117` |
| Time | HHMMSS | `143052` |
| Chain | kebab-case | `brandkit` |
| Random | 8 chars | `abc123de` |

### Refs (References)

Pass IDs, not data:

```json
{
  "refs": {
    "model": "gpt-4-turbo",
    "tone": "formal-corporate",
    "prompt": "brand-intro-v2",
    "colourPalette": "corporate-blue"
  }
}
```

Modules fetch their own data:
```javascript
const tone = await FileManager.get('tones', context.refs.tone);
```

### Trace (Execution Log)

```json
{
  "trace": [
    {
      "step": "Builders.scaffoldBrand",
      "status": "completed",
      "started": "2026-01-17T14:30:52Z",
      "completed": "2026-01-17T14:30:53Z"
    },
    {
      "step": "TokenManager.validateAndTrack",
      "status": "in_progress",
      "started": "2026-01-17T14:30:53Z"
    }
  ]
}
```

## Repo Access Rules

### Read Access

| Need | Tool | Pattern |
|------|------|---------|
| Know exact ID | FileManager | `FileManager.get('tones', 'formal-corporate')` |
| Need to search | Librarian | `Librarian.search('tones', { domain: 'business' })` |
| Need alternative | Librarian | `Librarian.recommend('tones', { exclude: [...] })` |

### Write Access

All writes go through Archivist → Writer:

```
Module output → Archivist.save() → validates → Writer.write() → disk
```

**Never write directly to repos** - always use Archivist.

### Cache Invalidation

When repo changes:
```javascript
await Archivist.save('tones', newEntry);
await Librarian.cache.invalidate('tones');  // Auto-called by Archivist
```

## Chain Execution Rules

### Step Dependencies

Steps in a chain run sequentially by default:

```json
{
  "steps": ["A", "B", "C"]  // A → B → C
}
```

### Parallel Execution

Use `parallel` wrapper for concurrent steps:

```json
{
  "steps": [
    "Setup.init",
    { "parallel": ["Legal.check", "Visual.process", "Voice.analyze"] },
    "Export.compile"
  ]
}
```

### Checkpoint Rules

Mark critical points for recovery:

```json
{
  "step": "AICaller.callWithRetry",
  "checkpoint": true
}
```

On failure after checkpoint, restart from checkpoint, not beginning.

## Version Control

### Schema Versioning

All schemas have versions:

```json
{
  "$schema": "tone.schema.json",
  "schemaVersion": "1.0"
}
```

### Migration on Version Change

When schema version changes:
1. Old entries still valid (backward compatible)
2. Migration script updates entries on access
3. Index rebuilt after migration

## Security Rules

### Never Store in Repos

| Data Type | Where Instead |
|-----------|---------------|
| API keys | .env + APIKeyManager |
| User passwords | Never (OAuth only) |
| OAuth tokens | Encrypted in AuthHandler |
| Secrets | APIKeyManager |

### Validation Required

All external input validated before processing:
- API responses → Parsers.validate()
- User input → Schema validation
- File uploads → Type checking
