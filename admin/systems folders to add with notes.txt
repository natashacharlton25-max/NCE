## 1. PostCreationQualityDecider
**Category:** `checks/` (Quality Decision Orchestration)

**Role:** Decision Engine for Quality Checks

**Intention:** 
Takes outputs from PostCreationCheckManager (GrammarCheck, SpellingCheck, LanguageCheck, LayoutCheck, VisualCheck, ValuesCheck) and decides what to do with issues:
- **Pass**: Content meets quality threshold → publish
- **Retry**: Specific section fails → regenerate that section
- **Manual Review**: Critical issues that can't auto-fix → flag for human
- **Salvage**: Partial content passes → keep what's good, regenerate failures

**Why Critical:**
- PostCreationCheckManager runs checks but doesn't decide outcomes
- Currently ambiguous who decides "is this content good enough?"
- Blocks automated content publishing workflow

**Key Responsibilities:**
- Evaluate check results against quality policies (per brand)
- Determine retry vs. salvage vs. manual review
- Trigger regeneration if needed (call TemplateEngine for failed sections)
- Track decision history (why was this salvaged? why manual?)
- Return final content or escalation

**Depends On:**
- PostCreationCheckManager (consumes check results)
- PolicyEngine (defines quality thresholds per brand)
- TemplateEngine (regenerates failed sections)
- FailureHandler (manages retry failures)
- Logger (audit trail of decisions)

**Feeds Into:**
- PublishingOrchestrator (final output decision)
- ContentSalvageModule (if partial salvage needed)

**Data In/Out:**
```javascript
// Input: Check results from PostCreationCheckManager
{
  jobId: string,
  content: object,
  checks: {
    grammar: { passed: boolean, issues: [] },
    spelling: { passed: boolean, issues: [] },
    language: { passed: boolean, issues: [] },
    layout: { passed: boolean, issues: [] },
    visual: { passed: boolean, issues: [] },
    values: { passed: boolean, issues: [] }
  }
}

// Output: Quality decision
{
  decision: "pass" | "retry" | "manual" | "salvage",
  content: object,          // final or partially fixed
  reason: string,
  failedSections?: string[],
  recommendedActions: string[],
  needsManualReview: boolean,
  escalationLevel: "none" | "warning" | "critical"
}
```

## 8. ContentSalvageModule
**Category:** `content/` or `orchestration/`

**Role:** Partial Content Recovery & Graceful Degradation

**Intention:**
When content generation fails mid-process, salvage what's complete rather than lose everything:
- Section 1 ✓, Section 2 ✓, Section 3 ✗ → Return sections 1-2, flag section 3 for retry
- Quality check fails on imagery but text passes → Keep text, flag images
- Provider timeout on 3rd variant → Return completed variants 1-2
- Format generation fails on Canva but succeeds on PDF → Publish PDF, retry Canva

**Why High Priority:**
- Prevents catastrophic loss of partial work
- Enables graceful degradation (publish what's done, retry what failed)
- Improves system resilience
- Reduces wasted computation

**Key Responsibilities:**
- Track what succeeded vs. failed at each pipeline stage
- Determine what's salvageable (complete sections, successful variants)
- Merge salvaged + new content on retry
- Flag salvaged content (note what's missing)
- Provide retry strategy for failed portions

**Depends On:**
- FailureHandler (understand failure types)
- PostCreationQualityDecider (decide salvage vs. regenerate)
- ContentMetadataManager (track what succeeded/failed)
- Logger (audit trail of salvage decisions)

**Feeds Into:**
- PublishingOrchestrator (salvaged content may be partial)
- FailureHandler (retry recommendations)

**Data In/Out:**
```javascript

  failed: {
    sections: [3],
    formats: ["pdf"]

// Output: Salvage decision

  action: "salvage" | "retry" | "abort",
```

## 10. GeneratedContentCatalog

**Intention:**

**Key Responsibilities:**

- Provide search interface (by theme, audience, quality, format)
- Group related content (variants, adaptations)
