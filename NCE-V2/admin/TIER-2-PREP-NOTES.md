- **Parsers — design direction: component-tree atoms + shared schema contract** *(worked out by Natasha across the 2026-05-24/25 sessions; informed by the website renderer briefing of 2026-05-25, see `docs/WEBSITE-RENDERER-STATE-2026-05-25.md`; durably captured 2026-05-25)*

  *The dilemma.* Freeform AI output cannot be trusted to be structurally consistent — parsing it after the fact is inference and guesswork. A fully rigid pre-determined JSON structure leaves the AI no room to compose. Neither extreme is acceptable.

  *The resolution — constrained creativity over a fixed atom set.* The AI is given a fixed set of allowed structural "atoms" and composes freely within that vocabulary; output is born tagged. Structural typing moves to **generation time**; Parsers does not do freeform structural inference.

  *What the website briefing established.* The website renderer already defines this vocabulary concretely: a single component registry (17 registered components as of 2026-05-25 — Page, Section, Grid, Heading, Text, etc.), consumed as a recursive component tree where each node names a registered `component` and carries flat props. The atom set is therefore **not a greenfield invention** — it is effectively the website's component registry, and the master set is **global** (one shared vocabulary, a tree, not per-content-type sets). Note: 8 of 17 component schemas were mid prop-rename audit at the briefing date — the vocabulary is actively changing.

  *Effect on Parsers' role.* Parsers shrinks to a **deterministic check**: every node's `component` is a member of the registered set, and its props satisfy that component's schema. This is a hard fail at NCE-V2 generation time. It is the **only validation NCE-V2's generated output ever receives in production**: the website's validators (DEV-mode renderer check, build-time `validate-data.ts`) run at build-time over the website's own `src/data/` files and do not fire on R2-fetched content at production request-time. The website renders an unregistered component as nothing — a silent failure — so NCE-V2 must catch malformed output before JSON reaches R2. The Parsers gate is not redundant; it is the sole gate.

  *Schema ownership — Natasha's lean.* A **shared component-schema source**: one canonical set of schema definitions consumed by both NCE-V2 and the website, neither holding a private copy. This is the drift fix — same pattern as the colour-math shared-module decision. A **shared validator** is viable: NCE-V2 content generation is JS/TS on Cloudflare Workers and the website is JS/TS on Cloudflare, so the website's `schema-validator.ts` logic can run on both sides — no language/runtime barrier. (Validator portability is a design inference from shared runtime, not a briefing-stated fact.)

  *Open sub-questions — for the Parsers boundary review.* (1) Where the shared schemas physically live — extracting them from the website repo into a neutral home is a **cross-repo coordination task, actioned in the website project, not from NCE-V2**. (2) Change-governance between the two consumers. (3) Versioning, given the in-progress prop-rename audit. (4) Schema delivery to the shared validator — build-time disk read (website) vs bundled/fetched at Worker runtime (NCE-V2); a packaging detail, not a blocker. (5) Subsetting at use level — whether a given content-type uses the full atom palette or a subset — is a separate template-level decision, not within scope here.

- **Parsers — boundary review finding (2026-05-25): `system/Parsers` does not survive as conceived.** Reviewed against v2 design + the captured atom-list direction (see Parsers atom-list bullet above); v1 `ROLE-MATRIX.md` Parsers entry (`parseJSON`, `parseAIResponse`, `validateSchema`, `extractContent`) and v1 `system/Parsers` placeholder used as **orientation only, not as v2 commitments**. Three confirmed sub-findings:
  1. **Parse/extract half — dead, no v2 job.** v1 Parsers' core job was structural inference: take a freeform AI blob and extract structure (`parseJSON`, `parseAIResponse`, `extractContent` — 3 of its 4 ROLE-MATRIX functions). The captured atom-list direction kills this: generation produces an already-tagged component tree, so there is no freeform blob to infer structure from. Same husk shape as PythonRunner / DatabaseHandler / FailureHandler — a v1 role the v2 design eliminated.
  2. **Validate half — belongs to the shared validator, not a bespoke `system/Parsers`.** The surviving `validateSchema` need is served by the **shared component-schema validator** from the schema-ownership lean (see atom-list bullet above), NOT by a reimplementation in `system/`. v1 evidence reinforces this: validation was already duplicated in v1 (both Parsers and FileManager validated against schemas) — the exact drift a single shared validator eliminates. Reimplementing it in `system/Parsers` would re-create that drift.
  3. **Broader validation need — a coordinated multi-checker stage; deferred to its own review.** *(Recorded from Natasha's account; this development was worked out by Natasha and is captured here from that account, not co-derived in the current session window — provenance noted honestly per CLAUDE.md §11.)* The real surviving validation need is bigger than a thin wrapper: a **coordinated multi-checker validation stage** — spelling, grammar, brand/values, safeguarding, structural — with a **manager** coordinating which checks a given job requires, and failures routed to the failure handler. (v1's ContentManager coordination pattern is orientation for the *shape*, not a v2 commitment.) Where that manager lives, which systems own which individual checks, and the failure-handoff contract are **its own dedicated review** — explicitly deferred, explicitly NOT specced as part of Parsers, and explicitly not a `system/` slot by default.

  **Headline:** parse-half dead, validate-half → shared validator, broader validation → its own coordination-layer review. None of it survives as a `system/Parsers` subsystem. Parsers is the third `system/` slot not to survive cleanly (see `system/`-coherence note above).
