## Core Principles

### 1. Implement, Don't Decide

If you encounter something not in the specs:

```
❌ WRONG: Invent a solution and implement it
❌ WRONG: Assume what was probably intended
❌ WRONG: "Fix forward" by adding functionality

✅ RIGHT: STOP
✅ RIGHT: Document the gap in a DEVIATION-REPORT
✅ RIGHT: Ask for clarification or approval
✅ RIGHT: Only then continue
```

### 2. Shared Types Are Mandatory

All data shapes must flow from a single source:

```
TYPES.md → shared-types/ → Backend AND Frontend
```

**Never:**
- Duplicate type definitions
- Create types that aren't in TYPES.md
- Let backend and frontend define types separately

### 3. Backend-First (Default)

Standard execution order:
1. Phases 43-47: Backend complete and verified
2. Phases 48-53: Frontend using real backend

**Why:** Prevents "fantasy APIs" where frontend assumes endpoints that don't exist.

### 4. Performance Constraints Are Binding

DESIGN-SYSTEM.md specifies performance targets. You must meet them:
- Bundle size limits
- Core Web Vitals targets
- Loading time requirements

If you cannot meet a constraint, document why in DEVIATION-REPORT.

### 5. Verification Phases Never Collapse

Even if you combine other phases for speed:
- **Phase 47** (Backend Verification) — Must happen before frontend
- **Phase 53** (Implementation Verification) — Must happen before handoff

### 6. Generated Artifacts Are Read-Only

Generated files must not be manually edited:

| Generated By | Output | Edit Rule |
|--------------|--------|-----------|
| ORM/Schema | Database types, client | Edit schema/config, regenerate |
| API codegen | Type definitions | Edit source spec, regenerate |
| Build tools | Compiled output | Never edit |

**If a generated file needs changes:**
1. Change the source (schema, config, spec)
2. Regenerate
3. Verify output

**Never:** Edit generated files directly. This causes drift and breaks the single source of truth.

## When Things Go Wrong

### Spec is Ambiguous

```
1. Check related specs (might be clarified elsewhere)
2. Check PASS-NOTES.md for context
3. If still unclear:
   - Document the ambiguity
   - Make a reasonable assumption
   - Note the assumption in IMPLEMENTATION-LOG.md
   - Flag for review in Phase 53
```

### Spec Seems Wrong

```
1. DO NOT "fix" the spec by implementing differently
2. Create DEVIATION-REPORT:
   - What spec says
   - Why it seems wrong
   - What you would implement instead
3. Wait for approval
4. Only then implement the approved approach
```

### Missing Information

```
1. Check all input documents
2. Check ARCHITECTURE-PATTERNS.md for conventions
3. If truly missing:
   - STOP implementation of that feature
   - Document in DEVIATION-REPORT
   - Continue with other features
   - Return when information is provided
```

### Performance Issue

```
1. Implement per spec first
2. Measure actual performance
3. If constraint not met:
   - Document in IMPLEMENTATION-LOG.md
   - Try optimization within spec
   - If still not met, file DEVIATION-REPORT
```

## Code Standards

Follow ARCHITECTURE-PATTERNS.md for project-specific patterns.

### Universal Standards

**File Size:**
- Target ~300 lines per file
- Hard limit ~500 lines
- If exceeding, split the file

**Naming:**
| Type | Convention | Example |
|------|------------|---------|
| Components | PascalCase | `ProjectCard.svelte` |
| Functions | camelCase | `formatDate()` |
| Constants | UPPER_SNAKE | `MAX_RETRY_COUNT` |
| Types | PascalCase | `ProjectResponse` |
| Files (components) | PascalCase | `ProjectCard.svelte` |
| Files (utilities) | camelCase | `dateUtils.ts` |
| Folders | kebab-case | `project-settings/` |

**Comments:**
- Comment "why", not "what"
- Code should be self-documenting for "what"
- Complex logic gets explanatory comments

**Error Handling:**
- Use patterns from ERROR-CODES.md
- Never swallow errors silently
- Always return structured error responses

## Commit Discipline

### Commit Granularity

One commit per logical unit:
- ✅ "Add User model and migration"
- ✅ "Implement login endpoint"
- ✅ "Add ProjectCard component"
- ❌ "WIP"
- ❌ "Fix stuff"
- ❌ "Implement entire backend"

### Commit Message Format

```
type(scope): description

- Detail 1
- Detail 2

Refs: PHASE-XX
```

**Types:** feat, fix, refactor, docs, test, chore

**Examples:**
```
feat(auth): implement login endpoint

- Add POST /auth/login route
- Add JWT token generation
- Add password validation

Refs: PHASE-46

---

fix(api): correct error response format

- Match ERROR-CODES.md structure
- Add error_code field to all responses

Refs: PHASE-45
```

### Commit Audit

Phase 53 includes commit history review:
- Commits should tell the story of implementation
- Each phase should have clear commit(s)
- No "fixup" commits that hide history
